# Assess Active Module

First read root AGENTS.md in full and remain in LEARNER_TUTOR mode. Confirm exactly one active FROZEN_FOR_ATTEMPT module and its unchanged version. Do not edit source, tests, contract, or thresholds.

Treat a learner statement that they are ready for assessment, want to complete the active module, or want certification as invocation of this workflow. Also enter this workflow automatically when tutor review has established the readiness conditions in AGENTS.md. No exact prompt or private-assessor command is required from the learner.

Read assessment.yaml and all module plans. Run `./support/bin/apprentice design-status` and confirm required design/reflection before any executable assessment or commit recommendation. Any unanswered staged question blocks final assessment. If a `BEFORE CODE` answer must be reconstructed because the earlier curriculum or tutor allowed implementation first, record that timing limitation without penalizing the learner and require oral confirmation; do not represent it as contemporaneous pre-code evidence. Then confirm declared public gates. Identify the scoped learner-candidate paths: the active `DESIGN.md`, declared learner-owned implementation paths, and any required learner-authored tests. Inspect Git status. If any candidate path differs from `HEAD`, give `COMMIT CANDIDATE NOW` guidance from `prompts/learner/next-step.md` and pause final private assessment until the learner creates the commit; do not commit for them. Record the candidate hash and verify assessed learner files match it.

Run `./support/bin/apprentice assess` without an ID; it must derive the sole frozen module, run the public build/test/static gates, and then invoke the configured private assessor. Use only its category-level output. For a required mutation gate, require one explicit `MU-N KILLED PT-N` line for every mutant declared in MUTATION_PLAN.md followed by `MUTATION-GATE PASS`. These lines attest trusted assessor calibration against faulty reference candidates; do not describe them as mutations of learner code. Never locate, open, search, or quote private assessment files. If the bridge is unavailable, omits an attestation, reports NOT RUN, or exits nonzero, retain the relevant gate as unresolved and do not certify.

Run any other declared public property, benchmark, performance, memory, integrity, and manual checks not covered by the command. Inspect design/reflection and conduct the required oral defense; do not certify on executable evidence alone. If learner code changes, require a new candidate commit and rerun affected evidence. Write an assessment report from the template with the immutable candidate hash, per-gate evidence, unresolved risks, hint history, certification tier, and precise required work. Update progress only with demonstrated evidence. After certification, report `COMMIT EVIDENCE SEPARATELY` for reports/progress/activation changes. Basic examples alone cannot pass the module.
