# Rolling Learning-Block Authoring Lifecycle

## 1. Evidence review

Read learner-profile, weaknesses, journal, review schedule, completed modules, recent assessment/benchmark reports, hint history, and boss review. Distinguish passing tests from demonstrated explanation and transfer.

## 2. Proposal only

Propose four to eight modules and one cumulative challenge. For each record capability, prerequisites, likely weak approach, invariants, edge classes, complexity/memory risks, motivating future abstraction, and three recurrence contexts. Do not create source or concrete tests yet.

## 3. Six independent reviews

Run and record separate specification, property-testing, adversarial, performance/memory, Haskell-semantics, and pedagogy/solution-leakage reviews. Each produces findings and required revisions. Review existing content skeptically; file existence is not approval.

## 4. Revision

Record each accepted, rejected, or deferred finding and why. Recheck block dependencies and boss theory budget. A rejected finding needs a concrete rationale.

## 5. Implementation

Create exact contracts, structured design questions, stubs, public tests/properties, visible private category plans, metadata, realistic mutants, performance/memory plans, reflection, changelog, and revisit plan. Follow `DESIGN_QUESTIONS.md` and `MODULE_GENERATION_STANDARD.md`. Never create visible solutions.

## 6. Pre-attempt audit

Map every behavior, invariant, edge, law, complexity/memory/strictness/totality requirement, forbidden shortcut, design question, and extensibility requirement to meaningful assessment paths. Verify DQ structure/answerability/leakage and public/private generation conformance. “Manual review” must name what evidence the reviewer seeks. Repair unexplained gaps.

## 7. Readiness and freeze

Mark future modules AUDITED_READY after their block audit. Immediately before activation, refresh against toolchain and learner evidence, then freeze exactly one module with version and date/commit. Public material becomes the attempt contract.

## 8. Post-module adaptation

Update demonstrated abilities, evidence-backed weaknesses, confidence, recurrence, reinforcement, and revisit timing. Do not retroactively lower the trial.

## 9. Post-boss review

Evaluate transfer, ceremonial abstractions, scale-only failures, malformed-input/resource/concurrency failures, and roadmap timing. Use findings to propose the next block.

Later phases remain broad until Step 1 makes detailed design responsible.
